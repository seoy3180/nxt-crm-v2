# NXT CRM v2 — 제품 요구사항 문서 (PRD)

---

# Part 1. PRD

## 1. 개요

- **목적**: 기존 NXT CRM v1의 5개 도메인(고객, 계약, 매출, MSP, 교육)을 동일한 기능 범위로 재구축하되, 구조적 품질(테스트, 타입 안전성, 에러 처리, 보안, 성능)을 근본적으로 개선한다.
- **대상 사용자**: 사내 직원 (staff, team_lead, admin, c_level) — 약 20~50명 규모
- **배경**: v1은 프로토타입 수준으로 빠르게 구축되어 다음 문제를 가지고 있다:
  1. 테스트 완전 부재 → 회귀 버그 위험
  2. RLS 정책이 모든 인증 사용자에게 전체 접근 허용 → 보안 취약
  3. 타입 안전성 부족 (`as unknown as`, `any` 타입 남용)
  4. 에러 처리 일관성 없음 → 사용자 혼란
  5. 페이지네이션 없음 → 데이터 증가 시 성능 저하
  6. 컴포넌트 과대화 (200~300줄) → 유지보수 어려움
  7. API 검증 부분적 (Zod 일부만 적용)
  8. DB 마이그레이션 단일 파일 → 스키마 변경 추적 불가
  9. 문서화 부족
- **성공 지표**:
  - 테스트 커버리지 80% 이상 (단위 + 통합)
  - Lighthouse 성능 점수 90+ (데스크탑)
  - API 응답 시간 p95 < 500ms
  - 타입 에러 0건 (strict mode)
  - 주요 유저 플로우 E2E 테스트 100% 커버

## 2. 스코프

### In Scope
- 고객 관리 CRUD (계층 구조 2단계 제한, 비즈니스 타입, 연락처)
- MSP 계약 파이프라인 (4단계 칸반 + 테이블)
- 교육 계약 파이프라인 (5단계 칸반 + 테이블)
- MSP 전용 대시보드 및 관리 기능
- 교육 운영 관리 (확장 필드: 중식/간식 제공여부, 계약인원, 모집인원 등)
- 매출 분석 (연간, 팀별, 월별, 타입별) — 역할별 접근 제한
- 역할별 맞춤 대시보드 (staff: 작업 중심, team_lead: 팀 매출, admin/c_level: 전사)
- 글로벌 검색 (Cmd+K) — PostgreSQL FTS + pg_trgm
- 사용자 인증 및 프로필 관리
- 역할 기반 접근 제어 (RBAC) — 팀 격리 정책
- v1 데이터 마이그레이션 (수십~수백 건, v2 완성 후 SQL 스크립트로 일괄 이전)
- Slack 통합 (v1 수준 유지: 커맨드 + 웹훅 + 알림) — Phase 3에서 구현

### 팀 구조
- **MSP팀**: MSP 계약/고객 관리
- **교육팀**: 교육 계약/운영 관리
- **개발팀**: Phase 3에서 상세 정의 (DB에 type='dev' 예약)

### Out of Scope (v2 이후)
- 모바일 전용 UI (반응형으로 최소 지원만)
- 외부 고객 포탈
- AI 기반 매출 예측
- 멀티 테넌시
- 감사 로그 (Audit Log) 전용 화면 — DB 레벨 기록만 포함
- Storybook (테스트에 집중, 필요 시 후속 추가)

## 3. 기술 스택 (v2)

| 영역 | v1 | v2 (계획) |
|------|-----|-----------|
| Framework | Next.js 15 + React 19 | Next.js 15 + React 19 (동일) |
| DB/Auth | Supabase (PostgreSQL + Auth) | Supabase (**별도 계정 + 새 프로젝트**, RLS 강화) |
| DB 모델 | 분리 (contracts_msp, contracts_tt) | **통합 모델** (contracts + 타입별 확장 테이블) |
| State | TanStack React Query 5 | TanStack React Query 5 (동일) |
| UI | shadcn/ui + Tailwind CSS 4 | shadcn/ui + Tailwind CSS 4 (동일) |
| Validation | Zod (부분 적용) | Zod (전체 적용) |
| Testing | 없음 | Vitest + Playwright |
| Type Safety | 느슨 | strict mode + 생성 타입 |
| Search | ILIKE | **PostgreSQL FTS + pg_trgm** (GIN 인덱스) |
| Rate Limiting | 없음 | **AWS API Gateway** 내장 throttling (분당 60회) |
| Rendering | SSR (Server Components) | **CSR** (S3 static export) — 사내 도구이므로 SEO 불필요 |
| 배포 | - | **AWS**: S3 + Route53 (프론트) / API Gateway + Lambda 또는 ECS Docker (API) / ECR (이미지) |

## 4. 기능 요구사항

### 도메인 1: 인증 및 사용자 관리

#### FR-001: 이메일/비밀번호 로그인
- **설명**: Supabase Auth를 통한 이메일/비밀번호 로그인. 로그인 성공 시 대시보드로 리다이렉트.
- **수락 기준**:
  - Given: 사용자가 로그인 페이지에 있다
  - When: 올바른 이메일/비밀번호를 입력하고 로그인 버튼을 클릭한다
  - Then: 대시보드(`/dashboard`)로 리다이렉트된다
  - Given: 잘못된 비밀번호를 입력했다
  - When: 로그인 버튼을 클릭한다
  - Then: "이메일 또는 비밀번호가 올바르지 않습니다" 메시지가 표시된다

#### FR-002: 세션 관리 및 자동 갱신
- **설명**: Supabase 세션 토큰 자동 갱신. 만료 시 로그인 페이지로 리다이렉트. 원래 URL 보존.
- **수락 기준**:
  - Given: 사용자의 세션이 만료되었다
  - When: 인증이 필요한 페이지에 접근한다
  - Then: 로그인 페이지로 리다이렉트되며, 로그인 후 원래 페이지로 복귀한다

#### FR-003: 사용자 프로필 관리
- **설명**: 이름, 이메일(읽기전용), 비밀번호 변경. 프로필 이미지 업로드는 Out of Scope.
- **수락 기준**:
  - Given: 사용자가 프로필 페이지에 있다
  - When: 이름을 변경하고 저장 버튼을 클릭한다
  - Then: "프로필이 업데이트되었습니다" 토스트가 표시되고, 사이드바의 이름이 즉시 반영된다

#### FR-004: 역할 기반 접근 제어 (RBAC)
- **설명**: 4단계 역할(staff, team_lead, admin, c_level)에 따른 기능 접근 제어. RLS 정책으로 DB 레벨 보호. **팀 격리 정책** 적용.
- **수락 기준**:
  - Given: staff 역할의 사용자가 로그인했다
  - When: 관리자 전용 기능(사용자 관리)에 접근한다
  - Then: "접근 권한이 없습니다" 메시지와 함께 대시보드로 리다이렉트된다
  - Given: admin 역할의 사용자가 로그인했다
  - When: 사용자 관리 페이지에 접근한다
  - Then: 전체 사용자 목록과 역할 변경 기능이 표시된다
- **사이드바 섹션 접근 규칙** (디자인은 통일, 코드에서 역할별 필터링):

| 섹션 | staff | team_lead | admin | c_level |
|------|-------|-----------|-------|---------|
| **NXT** (대시보드, 고객, 계약, 매출) | ❌ | ❌ | ✅ | ✅ |
| **MSP** (대시보드, 고객, 계약, 연락처) | MSP팀만 | MSP팀만 | ✅ | ✅ |
| **EDU** (대시보드, 계약, 운영) | 교육팀만 | 교육팀만 | ✅ | ✅ |
| **DEV** | 개발팀만 | 개발팀만 | ✅ | ✅ |

- **역할×기능 접근 매트릭스**:

| 기능 | staff | team_lead | admin | c_level |
|------|-------|-----------|-------|---------|
| 팀 대시보드 (MSP/EDU) | 소속 팀 (매출 제외) | 소속 팀 (매출 포함) | 전체 | 전체 |
| 고객 조회 | 소속 팀 계약 관련만 | 소속 팀 전체 | 전체 | 전체 |
| 고객 등록/수정/삭제 | 소속 팀 관련만 | 소속 팀 전체 | 전체 | 전체 |
| 계약 조회 | 소속 팀 배분 계약만 | 소속 팀 전체 | 전체 | 전체 |
| 계약 등록/수정/삭제 | 소속 팀만 | 소속 팀 전체 | 전체 | 전체 |
| 칸반 드래그 (단계변경) | 소속 팀 계약만 | 소속 팀 전체 | 전체 | 전체 |
| **NXT 대시보드 (전사)** | **❌** | **❌** | **✅** | **✅** |
| **매출 분석 (전사)** | **❌** | **❌** | **✅** | **✅** |
| **매출 분석 (팀별)** | **❌** | **소속 팀만** | **전체** | **전체** |
| 사용자 관리 | ❌ | ❌ | ✅ | ❌ |
| 글로벌 검색 | 소속 팀 범위 | 소속 팀 범위 | 전체 | 전체 |
| 프로필 수정 | 본인만 | 본인만 | 본인만 | 본인만 |

### 도메인 2: 고객 관리

#### FR-010: 고객 목록 조회
- **설명**: 전체 고객 목록을 테이블 형태로 표시. 검색, 타입 필터, 페이지네이션 지원.
- **수락 기준**:
  - Given: 고객이 50건 이상 등록되어 있다
  - When: 고객 목록 페이지에 접근한다
  - Then: 페이지당 20건씩 페이지네이션되어 표시된다
  - Given: 검색어 "넥스트"를 입력한다
  - When: 300ms 디바운스 후 검색이 실행된다
  - Then: 고객명에 "넥스트"가 포함된 결과만 필터링된다

#### FR-011: 고객 등록
- **설명**: 고객명(필수), 고객 유형(univ/corp/govt/asso/etc), 등급(A~F), 비즈니스 타입(msp/tt/dev), 부모 고객, 메모를 입력하여 고객을 등록. 고객 ID 자동 생성(UNIV-001 등).
- **수락 기준**:
  - Given: 사용자가 고객 등록 폼에 있다
  - When: 고객명 "테스트 대학교", 유형 "대학교"를 입력하고 저장한다
  - Then: "UNIV-XXX" 형식의 ID가 자동 생성되고, 고객 상세 페이지로 이동한다
  - Given: 고객명을 입력하지 않았다
  - When: 저장 버튼을 클릭한다
  - Then: "고객명을 입력해주세요" 에러 메시지가 필드 하단에 표시된다

#### FR-012: 고객 상세 조회 및 수정
- **설명**: 고객 기본 정보, 연락처 목록, MSP/교육 전용 정보, 관련 계약 목록, 하위 고객 목록을 한 화면에서 조회 및 수정.
- **수락 기준**:
  - Given: 고객 상세 페이지에 있다
  - When: 고객명을 수정하고 저장한다
  - Then: "고객 정보가 업데이트되었습니다" 토스트가 표시된다
  - Given: 해당 고객에 MSP 비즈니스 타입이 있다
  - When: 상세 페이지를 조회한다
  - Then: MSP 전용 탭(산업분야, 기업규모, 메모, 태그)이 표시된다

#### FR-013: 고객 계층화 (부모-자식 관계)
- **설명**: 고객 등록/수정 시 부모 고객을 지정하여 계층 구조 형성. 상세 페이지에서 하위 고객 목록 표시.
- **수락 기준**:
  - Given: 고객 등록 폼에서 부모 고객을 선택한다
  - When: 저장한다
  - Then: 부모 고객 상세 페이지의 하위 고객 목록에 새 고객이 표시된다

#### FR-014: 연락처 CRUD
- **설명**: 고객별 다수의 연락처(이름, 이메일, 전화, 직책, 역할, 대표 여부) 관리. 인라인 추가/수정/삭제(소프트 삭제).
- **수락 기준**:
  - Given: 고객 상세 페이지의 연락처 섹션에 있다
  - When: "연락처 추가" 버튼을 클릭하고 이름 "홍길동"을 입력하여 저장한다
  - Then: 연락처 목록에 "홍길동"이 추가된다
  - Given: 연락처 삭제 버튼을 클릭한다
  - When: "정말 삭제하시겠습니까?" 확인 대화상자에서 확인을 누른다
  - Then: 해당 연락처가 목록에서 사라진다 (소프트 삭제)

#### FR-015: 고객 소프트 삭제
- **설명**: 고객 삭제 시 물리 삭제가 아닌 `deleted_at` 타임스탬프 설정. 관련 연락처도 함께 소프트 삭제.
- **수락 기준**:
  - Given: 고객 상세 페이지에서 삭제 버튼을 클릭한다
  - When: 확인 대화상자에서 확인을 누른다
  - Then: 고객 목록에서 해당 고객이 사라지고, "고객이 삭제되었습니다" 토스트가 표시된다

### 도메인 3: MSP 계약 관리

#### FR-020: MSP 계약 파이프라인 (칸반 뷰)
- **설명**: 4단계(계약전 → 계약완료 → 사업완료 → 정산) 칸반보드. 드래그앤드롭으로 단계 변경. 카드에 계약명, 고객명, 금액 표시. **역방향 이동 허용** (확인 모달 표시). 카드 클릭 시 **새 페이지(`/contracts/[id]`)로 이동**.
- **수락 기준**:
  - Given: MSP 계약 목록 페이지에서 칸반 뷰를 선택했다
  - When: "계약전" 카드를 "계약완료" 컬럼으로 드래그한다
  - Then: 계약 단계가 변경되고, 이력(contract_history)이 자동 기록된다
  - Given: "사업완료" 카드를 "계약완료"로 역방향 드래그한다
  - When: "이전 단계로 되돌리시겠습니까?" 확인 모달이 표시된다
  - Then: 확인 시 단계가 변경되고, 이력에 역방향 이동이 기록된다
  - Given: 칸반 카드를 클릭한다
  - When: `/contracts/[id]` 계약 상세 페이지로 이동한다
  - Then: 계약 전체 정보(기본정보, 팀 배분, 비용, 이력)가 표시된다

#### FR-021: MSP 계약 테이블 뷰
- **설명**: 칸반 대안으로 테이블 형식 조회. 단계별 필터, 검색, 페이지네이션 지원.
- **수락 기준**:
  - Given: MSP 계약 목록 페이지에서 테이블 뷰를 선택했다
  - When: "계약완료" 필터를 선택한다
  - Then: stage가 "contracted"인 계약만 테이블에 표시된다

#### FR-022: MSP 계약 등록
- **설명**: 계약명(필수), 고객(필수), 설명, 총 금액, 통화, 담당자, 연락처를 입력하여 계약 등록. 초기 stage는 "pre_contract". 고객의 business_types에 "msp" 자동 추가.
- **수락 기준**:
  - Given: MSP 계약 등록 폼에 있다
  - When: 계약명과 고객을 입력하고 저장한다
  - Then: "MSP-XXX" 형식의 ID가 자동 생성되고, 계약 상세 페이지로 이동한다
  - Then: 해당 고객의 business_types에 "msp"가 자동으로 추가된다

#### FR-023: MSP 계약 상세 조회 및 수정
- **설명**: 계약 기본 정보(금액, 담당자, 결제방식, 빌링 레벨, 단계 등) 조회 및 수정. **편집 UX는 두 가지 패턴**:
  - **상세 페이지**: 상단 `[수정]` `[삭제]` 버튼. 수정 클릭 → 모든 필드 편집 가능 → `[저장]` `[취소]`로 전환. 적용 대상: 고객 상세, 계약 상세, 모든 상세 페이지.
  - **테이블 목록**: 상단 `편집 모드` 토글 버튼. 켜면 다음 기능이 활성화:
    1. **셀별 인라인 편집**: 편집 가능한 셀에 연한 파란 보더 표시. 클릭 시 즉시 편집, 포커스 이동 시 자동 저장. 편집 불가 셀(고객명, 계약 수 등 계산값)은 보더 없이 표시.
    2. **컬럼 설정**: `컬럼 설정` 버튼으로 표시할 컬럼 선택/해제 가능. 사용자별 preferences에 저장.
    3. **컬럼 순서 변경**: 헤더의 드래그 핸들(⋮⋮)로 컬럼 순서를 드래그앤드롭으로 변경. 사용자별 preferences에 저장.
    - 편집 모드 ON 시 테이블 보더가 파란색으로 변경되어 편집 중임을 시각적으로 표시.
    - 적용 대상: MSP 고객 테이블, 계약 테이블 뷰, 연락처 테이블, 교육 운영 테이블.
  - 통합 `useInlineEdit` 커스텀 훅으로 테이블 인라인 편집 구현.
- **수락 기준**:
  - Given: 계약 상세 페이지에서 `[수정]` 버튼을 클릭한다
  - When: 필드들이 편집 가능 상태가 되고, 금액/담당자/단계 등을 수정한다
  - Then: `[저장]` 클릭 시 변경이 저장되고 "저장되었습니다" 토스트가 표시된다
  - Given: 계약 테이블 뷰에서 `편집 모드`를 켠다
  - When: 금액 셀을 클릭하여 새 금액을 입력하고 포커스를 이동한다
  - Then: 변경이 즉시 저장되고 "저장되었습니다" 토스트가 표시된다

#### FR-024: 계약 단계 변경 및 이력 추적
- **설명**: 계약 단계 변경 시 contract_history에 from_stage, to_stage, changed_by, note, timestamp 기록. 상세 페이지에서 이력 타임라인 표시.
- **수락 기준**:
  - Given: 계약 상세 페이지에서 단계 변경 버튼을 클릭한다
  - When: 새 단계를 선택하고 메모를 입력한다
  - Then: 단계가 변경되고, 이력 타임라인에 새 항목이 추가된다

#### FR-025: 팀별 매출 배분
- **설명**: 계약별로 여러 팀에 매출 비율(%)을 배분. 합계 100% 검증.
- **수락 기준**:
  - Given: 계약 상세 페이지의 매출 배분 섹션에 있다
  - When: 팀 A에 60%, 팀 B에 40%를 입력하고 저장한다
  - Then: 매출 배분이 저장된다
  - Given: 합계가 100%를 초과한다
  - When: 저장을 시도한다
  - Then: "매출 배분 합계가 100%를 초과합니다" 에러 메시지가 표시된다

#### FR-026: MSP 계약 소프트 삭제
- **설명**: 계약 삭제 시 관련 contract_teams, expenses도 함께 소프트 삭제. **정산 완료(settled) 계약은 삭제 불가** — 삭제 필요 시 관리자에게 문의 안내.
- **수락 기준**:
  - Given: 계약 상세 페이지에서 삭제 버튼을 클릭한다
  - When: 확인 대화상자에서 확인을 누른다
  - Then: 관련 팀 배분과 비용도 함께 소프트 삭제되고, 계약 목록으로 이동한다
  - Given: 정산 완료(settled) 상태의 계약 상세 페이지에 있다
  - When: 삭제 버튼이 비활성화되어 있다
  - Then: 호버 시 "정산 완료된 계약은 삭제할 수 없습니다. 관리자에게 문의하세요." 툴팁 표시

### 도메인 4: 교육 계약 관리

#### FR-030: 교육 계약 파이프라인 (칸반 + 테이블)
- **설명**: 5단계(제안 → 계약완료 → 운영중 → 운영완료 → 정산) 칸반보드 및 테이블 뷰. "미지정" 필터로 stage가 null인 항목도 조회 가능.
- **수락 기준**:
  - Given: 교육 계약 목록 페이지에 있다
  - When: "미지정" 필터를 선택한다
  - Then: stage가 null인 계약만 표시된다

#### FR-031: 교육 계약 등록
- **설명**: 프로젝트명(필수), 총 금액, 통화, 단계, 고객을 입력하여 등록. ID는 "CT{연도}{순번}" 형식 (예: CT2026001).
- **수락 기준**:
  - Given: 교육 계약 등록 폼에 있다
  - When: 프로젝트명 "AWS 클라우드 교육"을 입력하고 저장한다
  - Then: "CT2026XXX" 형식의 ID가 자동 생성된다

#### FR-032: 교육 계약 단계 변경 및 금액 수정
- **설명**: 칸반 드래그 또는 상세 페이지에서 단계 변경. 금액 인라인 편집.
- **수락 기준**:
  - Given: 교육 계약 상세에서 금액을 수정한다
  - When: 새 금액을 입력하고 저장한다
  - Then: 금액이 즉시 업데이트된다

### 도메인 5: 교육 운영 관리

#### FR-040: 교육 운영 목록 조회
- **설명**: 운영 ID, 계약명, 운영명, 시작/종료일, 총 시간, 신청인원, 실제인원, 교육장소, 대상기관을 테이블로 표시.
- **수락 기준**:
  - Given: 교육 운영 페이지에 접근한다
  - When: 운영 목록이 로드된다
  - Then: 관련 계약명이 함께 표시되고, 시작일 내림차순으로 정렬된다

### 도메인 6: MSP 전용 기능

#### FR-050: MSP 대시보드
- **설명**: MSP 고객 요약 카드(총 고객 수, 빌링 레벨 분포, 파이프라인 분포). 고객별 테이블(빌링 레벨, 크레딧 쉐어, 예상 MRR, 결제자, 영업 담당, 연락처 정보).
- **수락 기준**:
  - Given: MSP 대시보드 페이지에 접근한다
  - When: 페이지가 로드된다
  - Then: 요약 카드에 총 고객 수, 빌링 레벨별 분포가 표시된다

#### FR-051: MSP 인라인 필드 편집
- **설명**: MSP 테이블에서 개별 필드(빌링 레벨, 크레딧 쉐어, 결제자, 영업 담당 등)를 직접 편집. 화이트리스트 기반 필드 검증.
- **수락 기준**:
  - Given: MSP 테이블에서 빌링 레벨 셀을 클릭한다
  - When: "MSP15"를 선택한다
  - Then: 즉시 저장되고 셀이 업데이트된다

#### FR-052: MSP 고객 목록
- **설명**: MSP 비즈니스 타입이 있는 고객만 필터링하여 표시.
- **수락 기준**:
  - Given: MSP > 고객 목록 페이지에 접근한다
  - When: 페이지가 로드된다
  - Then: business_types에 "msp"가 포함된 고객만 표시된다

#### FR-053: MSP 연락처 관리
- **설명**: MSP 고객에 속한 연락처만 모아서 표시. 고객명, 이름, 이메일, 전화, 직책, 역할, 대표 여부.
- **수락 기준**:
  - Given: MSP > 연락처 페이지에 접근한다
  - When: 페이지가 로드된다
  - Then: MSP 고객의 연락처만 고객명과 함께 표시된다

#### FR-054: AWS 계정 관리
- **설명**: MSP 고객별 AWS 계정 ID 관리. 글로벌 검색에서 AWS 계정 ID로 검색 가능.
- **수락 기준**:
  - Given: MSP 고객 상세 페이지에 있다
  - When: AWS 계정 ID를 추가한다
  - Then: 계정이 저장되고, 글로벌 검색에서 해당 ID로 검색 가능해진다

### 도메인 7: 매출 분석

#### FR-060: 연간 매출 현황
- **설명**: 선택한 연도의 월별 매출 추이 차트. MSP/교육/개발 타입별 분류. 분기별, 반기별 합계.
- **수락 기준**:
  - Given: 매출 분석 페이지에서 2026년을 선택한다
  - When: 데이터가 로드된다
  - Then: 12개월 막대 차트와 분기별/반기별 합계 테이블이 표시된다

#### FR-061: 팀별 매출 분석
- **설명**: contract_teams의 percentage 기반으로 팀별 매출 합산. 막대 차트 + 순위 테이블.
- **수락 기준**:
  - Given: 팀별 매출 탭을 선택한다
  - When: 데이터가 로드된다
  - Then: 팀별 매출 합계가 내림차순으로 표시된다

### 도메인 8: 전사 기능

#### FR-070: 역할별 맞춤 대시보드
- **설명**: 역할에 따라 다른 위젯 구성을 표시하는 대시보드.
- **역할별 위젯 구성**:

| 위젯 | staff | team_lead | admin | c_level |
|------|-------|-----------|-------|---------|
| KPI: 총 고객 수 | 소속 팀 | 소속 팀 | 전사 | 전사 |
| KPI: 활성 계약 수 | 소속 팀 | 소속 팀 | 전사 | 전사 |
| KPI: 신규 고객 수 | 소속 팀 | 소속 팀 | 전사 | 전사 |
| KPI: 총 매출 | ❌ | 소속 팀 | 전사 | 전사 |
| 파이프라인 요약 | 소속 팀 | 소속 팀 | 전사 | 전사 |
| 월별 매출 추이 차트 | ❌ | 소속 팀 | 전사 | 전사 |
| 팀별 매출 차트 | ❌ | ❌ | 전사 | 전사 |
| 최근 활동 (계약 이력) | 소속 팀 | 소속 팀 | 전사 | 전사 |
| **MSP팀 전용**: 활성 고객 수, 최근 단계 변경 | ✅ (MSP팀) | ✅ (MSP팀) | ✅ | ✅ |
| **교육팀 전용**: 진행 중 운영, 예정 교육, 최근 등록 | ✅ (교육팀) | ✅ (교육팀) | ✅ | ✅ |

- **수락 기준**:
  - Given: staff(MSP팀) 역할의 사용자가 대시보드에 접근한다
  - When: 페이지가 로드된다
  - Then: 소속 팀 KPI(매출 제외) + 파이프라인 요약 + 최근 활동 + MSP 전용 위젯이 표시된다
  - Given: c_level 역할의 사용자가 대시보드에 접근한다
  - When: 페이지가 로드된다
  - Then: 전사 KPI 4개 + 차트 3개 + 최근 활동 + MSP/교육 전용 위젯이 모두 표시된다

#### FR-071: 글로벌 검색 (Cmd+K)
- **설명**: 고객, 연락처, 계약, AWS 계정을 동시에 검색. 최소 2자 입력 후 검색 실행. 결과 타입별 그룹핑. 키보드 탐색(↑↓) 및 Enter로 이동.
- **수락 기준**:
  - Given: 어느 페이지에서든 Cmd+K를 누른다
  - When: 검색 모달이 열린다
  - Then: 검색어 입력 시 고객/연락처/계약/AWS 계정 결과가 타입별로 그룹화되어 표시된다
  - Given: 검색어가 1자인 경우
  - When: 입력한다
  - Then: 검색이 실행되지 않고 "2자 이상 입력해주세요" 안내가 표시된다

#### FR-072: 사용자 환경설정
- **설명**: MSP 대시보드 컬럼 표시/숨김 설정. 사용자별 preferences 저장.
- **수락 기준**:
  - Given: MSP 대시보드에서 컬럼 설정 버튼을 클릭한다
  - When: "예상 MRR" 컬럼을 숨긴다
  - Then: 해당 컬럼이 테이블에서 숨겨지고, 다음 로그인 시에도 설정이 유지된다

# Part 2. 유저 플로우

## Primary Flow 1: 고객 등록 → 계약 생성

1. 사용자가 사이드바에서 "고객 관리"를 클릭한다
2. 고객 목록 페이지가 표시된다 (테이블 + 검색 + 필터)
3. "새 고객" 버튼을 클릭한다
4. 고객 등록 폼이 표시된다 (고객명 필드에 자동 포커스)
5. 고객명, 유형, 등급, 비즈니스 타입을 입력한다
6. "저장" 버튼을 클릭한다
7. 시스템이 고객 ID를 자동 생성하고, 고객 상세 페이지로 이동한다
8. 연락처 섹션에서 "연락처 추가"를 클릭하여 담당자를 등록한다
9. 사이드바에서 "계약 관리"로 이동하여 새 MSP 계약을 등록한다
10. **결과**: 고객 + 연락처 + MSP 계약이 연결된 상태로 생성됨

## Primary Flow 2: MSP 계약 파이프라인 관리

1. 사용자가 사이드바에서 "계약 관리"를 클릭한다
2. 칸반 뷰가 기본으로 표시된다 (4열: 계약전/계약완료/사업완료/정산)
3. "계약전" 컬럼의 계약 카드를 "계약완료"로 드래그한다
4. 시스템이 Optimistic UI로 즉시 이동시키고, 서버에 단계 변경 요청을 보낸다
5. 성공 시 "단계가 변경되었습니다" 토스트 표시, 이력 자동 기록
6. 카드를 클릭하여 상세 페이지로 이동한다
7. 금액, 빌링 레벨 등을 인라인 편집한다
8. 팀 배분 섹션에서 매출 비율을 설정한다
9. **결과**: 계약이 파이프라인을 통해 진행되고, 모든 변경 이력이 추적됨

## Primary Flow 3: 매출 분석

1. 사용자가 사이드바에서 "매출 분석"을 클릭한다
2. 현재 연도의 월별 매출 차트가 표시된다
3. 연도 드롭다운에서 다른 연도를 선택한다
4. 해당 연도의 월별/분기별/반기별 매출 데이터가 갱신된다
5. "팀별" 탭을 클릭한다
6. 팀별 매출 내림차순 차트와 테이블이 표시된다
7. **결과**: 사용자가 시기별/팀별 매출 현황을 파악함

## Alternative Flow

### AF-1: 고객 목록에서 직접 검색하여 상세 접근
- **분기 조건**: 고객명을 이미 알고 있는 경우
- **흐름**:
  1. 고객 목록 페이지에서 검색 필드에 고객명 입력
  2. 300ms 디바운스 후 필터링된 결과 표시
  3. 해당 고객 행 클릭 → 상세 페이지

### AF-2: 글로벌 검색으로 빠른 접근
- **분기 조건**: 현재 페이지와 무관하게 특정 리소스를 찾아야 하는 경우
- **흐름**:
  1. Cmd+K로 검색 모달 열기
  2. 고객명/연락처/계약명/AWS 계정 ID 입력
  3. 결과에서 키보드(↑↓) 또는 클릭으로 선택
  4. 해당 리소스의 상세 페이지로 이동

### AF-3: 테이블 뷰로 계약 필터링
- **분기 조건**: 특정 단계의 계약만 확인하고 싶은 경우
- **흐름**:
  1. 계약 목록에서 테이블 뷰 토글
  2. 단계 필터 드롭다운에서 "계약완료" 선택
  3. 필터링된 테이블 결과 확인

## Error Flow

### EF-1: 고객 등록 시 서버 에러
- **발생 조건**: API 서버 응답 실패 (500)
- **에러 메시지**: "고객 등록에 실패했습니다. 잠시 후 다시 시도해주세요."
- **복구 방법**: 입력 데이터는 폼에 유지됨. "다시 시도" 가능.

### EF-2: 칸반 드래그 시 네트워크 에러
- **발생 조건**: 단계 변경 API 호출 실패
- **에러 메시지**: "단계 변경에 실패했습니다. 이전 단계로 복원합니다."
- **복구 방법**: Optimistic update 롤백으로 카드가 원래 컬럼으로 복귀

### EF-3: 세션 만료 중 데이터 수정
- **발생 조건**: 인라인 편집 또는 폼 제출 시 401 응답
- **에러 메시지**: "로그인이 만료되었습니다. 다시 로그인해주세요."
- **복구 방법**: 로그인 모달 표시 → 로그인 성공 후 자동 재시도

### EF-4: 존재하지 않는 리소스 접근
- **발생 조건**: 삭제된 고객/계약의 URL 직접 접근
- **에러 메시지**: "요청하신 항목을 찾을 수 없습니다."
- **복구 방법**: 해당 목록 페이지로 이동하는 링크 제공

## 13. UI 설계 결정사항 (Phase 3)

### 사이드바
- 섹션 구분: NXT / MSP / EDU / DEV (구분선 + 라벨)
- 글로벌 검색바: 사이드바 상단, ⌘K 단축키 힌트
- 사용자 프로필: 사이드바 하단, 아바타 + 이름 + 소속/역할 + ⚙ 설정 아이콘
- 로고: 헥사곤 + N 아이콘 (배경 없음, 파란 아웃라인)
- RBAC 기반 섹션 표시/숨김 (디자인 통일, 코드에서 필터링)

### 고객 관리
- **고객 목록**: 트리뷰 (부모-자식 2단계), 컬럼 순서: 고객명 → 비즈니스 → 유형 → 등급 → 사내 담당자 → 고객사 담당자 → 계약 수
- **필터**: 고객 유형 ▼ + 비즈니스 ▼ + 검색 (한 행에 배치, 고객 추가 버튼은 오른쪽 끝)
- **고객 유형 옵션**: 대학교, 기업, 공공기관, 협회, 기타
- **등급**: A~E (5단계)
- **자식 고객 상세 탭**: 기본 정보 | 연락처 | MSP 정보 | 교육 정보 | 관련 계약 (하위 고객 탭 제거)
- **부모 고객 상세 탭**: 기본 정보 + 하위 고객 테이블 | 관련 계약 (자식 합산, "고객" 컬럼 추가). 헤더에 "상위 고객" 뱃지 표시. 연락처/MSP/교육 탭 없음.
- **기본 정보 탭**: 단일 카드 (고객사명/유형/등급 3열 + 비즈니스 타입/사내 담당자 + 메모). 고객 ID 미표시. 상위 고객 없으면 필드 숨김.
- **편집 패턴**: 카드별 타이틀 옆 ✏️수정 버튼 (보더 있음). 헤더에 수정/삭제 없음.
- **삭제 패턴**: 페이지 맨 하단 위험 영역 ("고객 삭제" + 설명 + 🗑삭제 버튼). 모든 탭에서 동일 위치.
- **연락처 탭 컬럼**: 이름 | 부서 | 직책 | 전화 | 이메일 (균등 배분, 5열)
- **MSP 정보 탭**: MSP 계약 요약 (읽기 전용, 맨 위) + MSP 정보 카드 (산업/규모/태그/메모) + MSP 계약 목록
- **교육 정보 탭**: 교육 계약 요약 (읽기 전용, 맨 위) + 교육 정보 카드 (속성 미정의, 빈 카드) + 교육 계약 목록
- **관련 계약 탭 컬럼**: 계약명 | 타입 | 단계 | 사내 담당자 | 고객사 담당자 | 금액

### 계약 관리 (NXT)
- **비즈니스 탭**: MSP | 교육 | 개발 (탭 선택에 따라 파이프라인 변경)
- **MSP 파이프라인**: 4단계 (계약전 → 계약완료 → 사업완료 → 정산)
- **교육 파이프라인**: 5단계 (제안 → 계약완료 → 운영중 → 운영완료 → 정산)
- **테이블 뷰**: 단계 필터 (전체 단계 기본) + 비즈니스 필터 + 검색
- **계약 등록**: 비즈니스 타입 선택 (MSP/교육/개발) → 공통 필드 + 비즈니스별 확장 필드
- **MSP 계약 등록 필드**: 기본(계약명, 고객, 금액, 통화) + MSP(빌링 레벨, 크레딧 쉐어, 예상 MRR, 결제자, 영업 담당, AWS 금액, 관리비 유무) + 설명
- **교육 계약 등록**: 기본 필드 + 교육 운영 동적 추가 (운영명, 교육 장소, 기간, 인원, 중식☑, 간식☑, 주강사▼, 보조강사1▼, 보조강사2▼). "+ 운영 추가" 버튼으로 차수 추가/삭제.
- **관리비 유무**: 라디오 토글 [있음|없음] (연결된 세그먼트 컨트롤)
- **중식/간식**: 체크박스 (체크=제공, 미체크=미제공, 텍스트도 상태 반영)
- **계약 상세 구조**: 공통 카드 (계약 정보) + 비즈니스별 확장 카드 (MSP 상세 / 교육 상세) + 오른쪽 (매출 배분 + 변경 이력)
- **교육 계약 상세**: 교육 상세 + 교육 운영 (차수) 테이블 + 사내 인력 배정 테이블
