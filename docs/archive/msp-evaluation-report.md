# NXT CRM v2 - MSP 섹션 종합 평가 리포트

평가일: 2026-04-09
평가 대상: NXT CRM v2 MSP 섹션 (대시보드, 고객, 계약, 연락처, 계약 상세/등록)

---

## 1. 일반 사용자 평가 (eval-public)

**평가자: 일반 사용자 관점 심사위원**

### 첫인상 (20점 만점)
**점수: 16/20**
- 깔끔하고 일관된 디자인 언어 사용 (zinc/blue 팔레트)
- 대시보드 진입 시 KPI 카드 3개로 핵심 수치 즉시 확인 가능
- 월별 매출 차트, 등급 분포, 파이프라인이 한 화면에 잘 정리됨
- 다만 KPI 카드가 3개뿐이라 약간 허전한 느낌

### 직관성 (20점 만점)
**점수: 16/20**
- 편집 모드 토글이 시각적으로 명확 (파란 테두리 + 토글 스위치)
- 칸반/테이블 뷰 전환이 URL에 유지되어 자연스러움
- 인라인 편집 시 변경된 셀이 파란색으로 표시되어 직관적
- 편집 모드에서 네비게이션 차단(EditMode Context)이 데이터 유실 방지에 효과적
- 연락처 추가 모달의 Combobox 검색이 편리함
- 다만 컬럼 설정 패널이 드롭다운 형태라 처음 찾기 어려울 수 있음

### 시각적 매력 (15점 만점)
**점수: 12/15**
- shadcn/ui + Tailwind 기반의 모던한 디자인
- 차트가 순수 CSS/div 기반이라 약간 투박함 (Recharts 등 차트 라이브러리 미사용)
- 배지 스타일링(MSP 등급, 단계)이 깔끔함
- 여백과 타이포그래피가 일관성 있음

### 가치 전달 (15점 만점)
**점수: 13/15**
- MSP 비즈니스에 필요한 핵심 정보(매출, 고객, 계약, 파이프라인)가 잘 구성됨
- 빠른 작업(새 계약, 새 고객, 검색)으로 주요 액션에 빠르게 접근 가능
- 최근 활동 피드로 팀 현황 파악 용이
- 매출 배분(팀별 %) 기능이 실무적으로 유용

### 불편함 (15점 만점)
**점수: 11/15**
- 대시보드의 차트가 인터랙티브하지 않음 (호버 툴팁, 클릭 드릴다운 없음)
- 고객 목록에서 클라이언트 상세 페이지로 이동하는데 `/clients/${id}` 경로를 사용하여 MSP 맥락에서 벗어남
- 연락처 페이지에 인라인 편집이나 컬럼 설정이 없어 다른 페이지와 기능 수준 불일치
- 계약 등록 후 `/contracts/${id}`로 이동하는데 MSP 계약이면 `/msp/contracts/${id}`여야 자연스러움

### 재방문 의사 (15점 만점)
**점수: 12/15**
- 사내 CRM으로서 필수적으로 사용해야 하는 도구이므로 재방문은 당연
- 인라인 편집과 칸반 뷰가 반복 사용에 편리
- 다만 필터/정렬 옵션이 제한적이라 데이터가 많아지면 불편할 수 있음

### 총점: 80/100 (A등급)

**총평:** "사내 CRM으로서 기본기가 탄탄하고, 인라인 편집과 칸반 보드가 일상 업무에 매우 편리합니다. 다만 차트의 인터랙티브성과 페이지 간 기능 일관성에서 아쉬움이 있습니다."

**[피드백]**
- 대시보드 차트에 호버 툴팁과 클릭 시 상세 조회 기능 추가 희망
- 연락처 페이지도 인라인 편집 지원 필요
- 검색/필터 기능 강화 (기간별, 금액 범위 등)

**[이랬으면 좋겠어요]**
- 모바일에서도 기본적인 조회가 가능했으면
- 계약 목록에서 다중 선택 후 일괄 단계 변경 기능
- 대시보드에 목표 대비 달성률 표시

## 2. UX 전문가 평가 (eval-ux)

**평가자: UX 전문가 (Nielsen 10가지 휴리스틱 기반)**

### H1. 시스템 상태 가시성 (10점)
**점수: 8/10**
- 편집 모드 ON/OFF 상태가 토글 + 파란 테두리로 명확히 표시됨
- 변경 건수 표시 (`3건 변경`)가 현재 상태를 잘 알려줌
- 저장 중 버튼 텍스트 변경 (`저장 중...`)으로 피드백 제공
- 로딩 상태에 Skeleton UI 적용
- **감점:** 칸반 드래그 중 대상 컬럼의 시각적 피드백(ring-2 ring-blue-200)은 좋으나, 드래그 시작 카드의 opacity 30%가 너무 투명

### H2. 실제 세계와의 일치 (10점)
**점수: 9/10**
- MSP 도메인 용어(MRR, 크레딧 쉐어, Payer, 청구 방식)가 실무에 맞게 사용됨
- 파이프라인 단계(사전계약, 계약진행, 정산완료 등)가 실제 업무 흐름과 일치
- 금액 표시가 한국어 관례(`₩ 1,234만`, `₩ 1.2억`)에 맞음
- 등급(MSP5, MSP10 등)이 실제 사내 용어와 일치

### H3. 사용자 통제와 자유 (10점)
**점수: 8/10**
- 편집 모드에서 취소 버튼으로 모든 변경 되돌리기 가능
- 계약 상세에서 수정/저장/취소 플로우가 명확
- beforeunload 이벤트로 미저장 데이터 유실 방지
- EditMode Context로 편집 중 네비게이션 차단
- **감점:** 편집 모드에서 개별 셀 변경 되돌리기(undo)가 없음. 전체 취소만 가능
- **감점:** 매출 배분 저장 후 되돌리기 불가 (soft delete 후 새로 insert)

### H4. 일관성과 표준 (10점)
**점수: 7/10**
- 고객/계약 목록 모두 같은 인라인 편집 A패턴 사용 - 일관적
- ColumnSettings 컴포넌트가 공용으로 재사용됨
- **감점 (심각도 2):** 연락처 페이지만 인라인 편집/컬럼 설정이 없어 불일치
- **감점 (심각도 2):** 계약 등록 후 라우팅이 `/contracts/${id}` (일반)와 `/msp/contracts/${id}` (MSP) 불일치
- **감점 (심각도 1):** 고객 목록 클릭 시 `/clients/${id}`로 이동하여 MSP 맥락 이탈

### H5. 에러 예방 (10점)
**점수: 8/10**
- 매출 배분 합계 100% 검증 (`total !== 100` 시 toast.error)
- 정산완료 상태 계약 삭제 차단 (ContractDeleteZone의 isSettled)
- Zod 스키마 기반 폼 검증 (contractCreateSchema)
- 편집 모드에서 '새 고객'/'새 계약' 버튼 비활성화
- **감점:** 금액 입력 필드에 inputMode="numeric"은 있지만 실제 숫자 이외 문자 입력 방지 없음

### H6. 인식을 통한 기억 최소화 (10점)
**점수: 8/10**
- Combobox로 고객/연락처 검색 (기억 대신 인식)
- Select 컴포넌트로 선택지를 표시
- 단계 배지에 라벨 표시 (코드값 대신)
- **감점:** 계약 등록 폼에서 이전 입력값 복원 기능 없음

### H7. 사용의 유연성과 효율성 (10점)
**점수: 7/10**
- 인라인 편집으로 목록에서 바로 수정 가능 (효율적)
- 칸반 드래그앤드롭으로 단계 변경 (빠른 조작)
- 컬럼 설정으로 개인화 가능 (user_preferences에 영구 저장)
- 대시보드 빠른 작업 + Cmd+K 검색
- **감점:** 키보드 단축키가 제한적 (Cmd+K 외에 편집 모드 토글 등 없음)
- **감점:** 벌크 작업(다중 선택 후 일괄 변경) 미지원

### H8. 심미적이고 미니멀한 디자인 (10점)
**점수: 8/10**
- 불필요한 장식 요소 없이 깔끔한 레이아웃
- 정보 밀도가 적절함
- 대시보드의 2컬럼 구조가 효과적
- **감점:** 대시보드 차트가 div 기반으로 다소 투박

### H9. 에러 인식, 진단, 복구 (10점)
**점수: 7/10**
- sonner toast로 성공/실패 피드백 제공
- ErrorState 컴포넌트로 에러 + 재시도 버튼 제공
- Zod 검증 에러가 필드별로 표시됨
- **감점:** 일괄 저장 시 부분 실패 핸들링이 없음 (Promise.all로 전체 성공/실패만)
- **감점:** 에러 메시지가 구체적이지 않음 ('저장 중 오류가 발생했습니다')

### H10. 도움말과 문서 (10점)
**점수: 5/10**
- 사내 CRM이라 외부 도움말은 불필요하지만
- 필드 설명(tooltip, placeholder)이 최소한
- 편집 모드 사용법에 대한 안내가 없음
- 온보딩이나 첫 사용 가이드 없음

### 총점: 75/100 (B등급)

**총평:** "인라인 편집 A패턴과 EditMode Context는 사내 CRM으로서 효율적인 UX 패턴입니다. 일관성(연락처 페이지 기능 누락, 라우팅 불일치)과 에러 핸들링 세밀화가 가장 시급한 개선 포인트입니다."

**[피드백]**
- 연락처 페이지의 인라인 편집/컬럼 설정 지원으로 일관성 확보 필요
- 라우팅 경로 통일 (MSP 계약 등록 후 MSP 경로로 이동)
- 개별 셀 undo 기능 추가
- 벌크 작업 지원

**[개선 제안]**
1. (심각도 3) 라우팅 일관성 수정 - 1일 이내
2. (심각도 2) 연락처 페이지 인라인 편집 추가 - 2-3일
3. (심각도 2) 에러 메시지 구체화 - 1일
4. (심각도 1) 도움말 툴팁 추가 - 1일
5. (심각도 1) 키보드 단축키 확장 - 2일

## 3. 기술 전문가 평가 (eval-tech)

**평가자: 기술 전문가 (성능, 보안, 접근성, 코드 품질)**

### 아키텍처 & 코드 품질 (25점)
**점수: 20/25**

**강점:**
- Service Layer 분리 (`contract-service.ts`)로 비즈니스 로직과 UI 분리
- React Query로 서버 상태 관리, staleTime 설정으로 불필요한 리페치 방지
- Zod 스키마 검증 (`contractCreateSchema`, `mspDetailSchema`)
- EditModeProvider Context로 전역 편집 상태 관리
- ColumnSettings를 공용 컴포넌트로 분리하여 재사용
- useColumnPreference 훅으로 user_preferences DB 연동 추상화

**약점:**
- `contract-service.ts` line 86: `const supabase = createClient()`가 모듈 레벨에서 호출됨 - SSR 환경에서 문제 가능. 함수 내부에서 호출해야 안전
- 계약 목록 페이지(`contracts/page.tsx`)가 730줄로 과도하게 큼. 인라인 편집 로직을 커스텀 훅으로 분리 필요
- 고객 목록도 유사한 패턴이 반복됨 - 인라인 편집 훅 공통화 가능
- `as Record<string, unknown>` 타입 캐스팅이 빈번 - 타입 안전성 저하

### 성능 (20점)
**점수: 16/20**

**강점:**
- 검색 디바운스 (`SEARCH_DEBOUNCE_MS`) 적용
- 테이블 페이지네이션 (20건 단위)
- employees 쿼리 `staleTime: 5 * 60 * 1000`으로 캐싱
- 칸반 뷰에서는 `pageSize: 100`으로 전체 로딩, 테이블에서는 20건 페이지네이션 - 적절한 분기

**약점:**
- 연락처 페이지에서 전체 contacts를 로딩 후 클라이언트 필터링 - 데이터 많아지면 성능 저하
- 고객 목록에서 contractCounts를 별도 쿼리로 전체 조회 후 Map으로 카운팅 - N+1은 아니지만 비효율
- 계약 목록 테이블 뷰에서 검색이 서버 쿼리가 아닌 클라이언트 필터링 (`debouncedSearch ? mapped.filter(...)`)
- 대시보드에서 5개 쿼리를 병렬 실행하지만, isLoading이 OR 조건이라 가장 느린 쿼리가 전체를 블로킹

### 보안 (20점)
**점수: 17/20**

**강점:**
- Supabase RLS 기반 데이터 접근 제어
- SECURITY INVOKER 뷰 사용 (명시적으로 언급됨)
- can_access_contract 함수로 계약 접근 권한 체크
- 검색 입력에 SQL Injection은 Supabase SDK가 방어
- soft delete 패턴 (`deleted_at`)으로 데이터 보존

**약점:**
- contract-service.ts에서 Supabase 클라이언트를 모듈 레벨에서 생성 - 인증 토큰 갱신 이슈 가능
- 일괄 저장 시 개별 에러가 Promise.all에 묻힘 - 부분 성공/부분 실패 시 데이터 불일치 가능
- 매출 배분에서 기존 데이터 soft delete 후 insert - race condition 가능

### 접근성 (15점)
**점수: 9/15**

**강점:**
- 테이블 행에 `tabIndex={0}`, `onKeyDown` (Enter) 지원
- aria-label 일부 사용 (`전역 검색 열기 (Cmd+K)`)
- 시맨틱 HTML (Table, TableHeader, TableBody 등) 사용

**약점:**
- 칸반 카드에 aria-label 없음
- 편집 모드 토글에 aria-pressed 없음
- 컬럼 설정 드롭다운에 aria-expanded 없음
- 스크린 리더로 인라인 편집 상태 인식 불가
- 색상만으로 상태 구분하는 곳이 있음 (합계 100%=green, >100%=red)

### 에러 핸들링 (10점)
**점수: 7/10**

**강점:**
- ErrorState 컴포넌트로 에러 + 재시도 구조화
- toast.error로 사용자 알림
- 변경이력 기록 실패 시에도 메인 저장은 성공 처리 (`.catch(() => {})`)

**약점:**
- try-catch에서 error 객체를 사용하지 않음 (`catch {}`) - 디버깅 어려움
- Promise.all 일괄 저장 시 부분 실패 복구 없음
- 네트워크 오프라인 감지 없음

### 코드 유지보수성 (10점)
**점수: 7/10**

**강점:**
- 상수 분리 (`MSP_STAGES`, `CREDIT_SHARE_OPTIONS` 등)
- 서비스 레이어와 UI 분리
- 공용 컴포넌트 (`ColumnSettings`, `KpiCard`, `ErrorState`)

**약점:**
- 인라인 편집 로직이 고객/계약 페이지에서 거의 동일하게 반복 (DRY 위반)
- formatAmount 함수가 3곳에서 각각 정의됨 - 유틸리티로 통합 필요

## 5. 까다로운 비평가 평가 (eval-critic)

**평가자: 까다로운 비평가**

### 이탈 포인트 분석 (20점)
**점수: 11/20**
- **치명적:** 계약 등록 후 `/contracts/${id}`로 이동 - MSP 계약을 등록했는데 일반 계약 경로로 보내버림. 사용자가 "어? 내 계약 어디 갔지?" 할 수 있음
- **치명적:** 대시보드 차트가 인터랙티브 제로. div로 그린 막대 차트를 보고 "이게 2026년 CRM이야?" 생각할 수 있음
- **심각:** 연락처 페이지만 편집 기능이 빠져있어 "이 페이지는 아직 미완성인가?" 인상
- **보통:** 테이블 뷰에서 클라이언트 검색(`mapped.filter`)인데 서버 검색인 것처럼 디바운스를 걸어둠. 데이터 20건 가져온 후 그 안에서만 검색하는 건데 사용자는 전체 검색인 줄 앎
- **보통:** 편집 모드에서 ESC로 셀 편집 취소는 되는데, 전체 편집 모드를 ESC로 끄는 건 안 됨

**점수: 13/20**

- 칸반 드래그앤드롭이 DnD Kit으로 구현되어 안정적
- 매출 배분의 100% 검증이 정확
- **감점:** 컬럼 설정 패널에 닫기 버튼이나 바깥 클릭 닫기가 없음 (코드상 onOpenChange만 있고 overlay가 없음)
- **감점:** 대시보드의 등급별 분포와 파이프라인이 숫자만 보여주고 클릭 시 상세로 이동하는 드릴다운이 없음

- **감점:** formatAmount가 3곳에서 각각 다르게 구현됨. `formatRevenue`는 또 별도로 있음. 이건 버그의 온상

### 엣지 케이스 처리 (20점)
**점수: 12/20**
- 빈 목록 상태 처리됨 ("MSP 고객이 없습니다", "등록된 MSP 계약이 없습니다")

- **감점:** 매출 배분에서 soft delete 후 insert 사이에 다른 사용자가 조회하면 빈 상태가 보임 (트랜잭션 없음)
- **감점:** 계약 삭제 시 contract_teams만 soft delete하고 contract_msp_details는 안 건드림
- **감점:** 숫자 입력 필드에서 "abc" 입력 시 `Number("abc")` = NaN이 DB에 들어갈 수 있음
- **감점:** 검색어에 SQL 와일드카드(`%`, `_`) 입력 시 예상치 못한 결과

1. **일반사용자(80점) vs 비평가(59점):** 21점 차이. 일반사용자는 "기본기 탄탄"으로 평가한 반면, 비평가는 데이터 정합성/동시 편집 등 "보이지 않는 문제"를 파고들어 낮게 평가. 실사용에서는 만족스럽지만 엣지 케이스에서 취약.
2. **비즈니스(79점) vs 비평가(59점):** 비즈니스 관점에서는 MSP 도메인 적합성을 높이 평가한 반면, 비평가는 경쟁사 대비 기능 부족을 지적. 사내 도구 vs 시장 제품의 기준 차이.
