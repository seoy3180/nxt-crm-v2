### 치명적 기술 이슈 (반드시 수정)
  - **반응형 레이아웃 미구현**: 사이드바 `w-60` 고정, `flex h-screen` 레이아웃에 브레이크포인트 없음. 모바일/태블릿 완전 사용 불가

  - **Content-Security-Policy 헤더 미설정**: XSS 공격에 대한 브라우저 레벨 방어 없음

### 권장 개선사항
  - CSP(브라우저가 허락된 코드만 실행하게 막는 보안 설정) 헤더를 next.config.ts의 headers()에 추가 (난이도: 보통)
  - robots.txt / sitemap.xml 추가 — 사내용이라 noindex 설정 권장 (난이도: 쉬움)
  - 시맨틱 HTML 개선: `<nav>` 내부 링크들에 `aria-current="page"` 추가, 주요 섹션에 `<section>`, `<article>` 활용 (난이도: 쉬움)
  - 접근성: ARIA 속성 22개만 발견 — 폼 요소 label 연결, 버튼 aria-label 보강 필요 (난이도: 보통)
  - Rate Limiting(같은 사람이 너무 빠르게 반복 요청하는 걸 막는 장치): 로그인 API에 시도 횟수 제한 없음 (난이도: 보통)

### 기술 스택 분석
  - 프레임워크: Next.js 15 (App Router) + React 19 + TypeScript 5
  - 상태관리: TanStack Query v5 (서버 상태) + React Context (인증)
  - UI: shadcn/ui + Radix UI + Tailwind CSS v4 + Lucide Icons
  - 백엔드: Supabase (PostgreSQL + GoTrue Auth + RLS)
  - 폼: React Hook Form + Zod v4 검증
  - 차트: Recharts v3
  - 특이사항: RLS(Row Level Security, 행 단위 접근 제어) 정책이 체계적으로 구현됨. `SECURITY DEFINER` 함수 + `search_path` 설정으로 보안 권고사항 준수. Zod 기반 클라이언트 검증이 잘 되어 있으나 서버 사이드 검증(Server Actions / API Routes) 레이어가 보이지 않음 — 클라이언트에서 직접 Supabase를 호출하는 구조

### 기술 개선 로드맵
  [즉시] 에러 페이지 구현
    - 뭘 해야 하는지: `src/app/not-found.tsx`, `src/app/error.tsx`, `src/app/global-error.tsx` 생성
    - 왜 급한지: 잘못된 URL 접근 시 사용자가 복구 불가, 빈 화면은 서비스 신뢰도 하락

  [즉시] CSP 헤더 추가
    - 뭘 해야 하는지: `next.config.ts`에 `headers()` 설정으로 Content-Security-Policy 추가
    - 왜 급한지: XSS(악성 스크립트 주입 공격) 방어의 기본. 사내용이라도 필수

  [단기 1-2주] 반응형 레이아웃
    - 뭘 해야 하는지: 사이드바를 모바일에서 드로어(서랍식 메뉴)로 전환, 그리드 레이아웃에 브레이크포인트(화면 크기별 전환점) 추가
    - 왜 필요한지: CRM은 외근 중 모바일 접근이 빈번. 현재 모바일 완전 사용 불가

  [단기 1-2주] 접근성(a11y) 보강
    - 뭘 해야 하는지: 폼 label 연결, 버튼 aria-label, 키보드 탐색 보장, 색상 대비 검증
    - 왜 필요한지: WCAG 2.1 AA 기준 미달. 스크린리더 사용자 접근 제한

  [중기 1-2개월] 서버 사이드 검증 레이어
    - 뭘 해야 하는지: Server Actions 또는 API Routes에 Zod 검증 + 권한 확인 추가
    - 왜 필요한지: 현재 클라이언트에서 직접 Supabase 호출. RLS가 있어 치명적이진 않지만, 방어 심도(Defense in Depth, 여러 겹의 보안 장치) 원칙상 서버 레이어 검증 추가 필요

### 기술 총평
  "기술 스택 선택과 아키텍처 설계는 적절합니다. Next.js 15 + React 19 + Supabase RLS 조합이 사내 CRM에 잘 맞고, TanStack Query 기반 데이터 페칭도 깔끔합니다. Zod 스키마 검증과 RLS 정책이 체계적인 점은 높이 평가합니다. 다만 반응형 미대응, 에러 바운더리 부재, CSP 미설정은 프로덕션 배포 전 반드시 해결해야 할 기술 부채입니다."

  vs **Salesforce/HubSpot**: 이기는 점 — 비용 0원(자체 개발), 한국어 UI, 커스텀 자유도 / 지는 점 — 기능 완성도, 모바일 앱, 이메일 연동, 자동화, 리포팅 전부. 규모가 다름
  vs **엑셀/구글 시트**: 이기는 점 — 검색, 관계형 데이터, 변경 이력, 권한 / 지는 점 — 엑셀은 이미 가지고 있고, 배우지 않아도 됨

  - **기술(69점) vs 비평가(52점) = 17점 차이**: 기술 전문가는 아키텍처와 보안 구조(RLS, Zod)를 높이 평가한 반면, 비평가는 "코드가 아무리 좋아도 데이터가 없으면 무의미"라는 입장. 둘 다 맞는 말이나, 현 시점에서는 비평가의 관점이 더 급함
  - **디자인 평가**: UX 전문가는 "shadcn 기반 일관된 디자인 시스템"을 강점으로 본 반면, 비평가는 계약 상세 레이아웃 깨짐(세로 텍스트)을 "학생 프로젝트 수준"으로 평가. 기본 틀은 좋으나 세부 QA 부족

### [종합 개선 우선순위]
  1순위: **계약 금액 데이터 입력 + 일괄 입력 도구** (4명 공통 지적) — 서비스 존재 이유
  2순위: **모바일 반응형 레이아웃** (5명 공통 지적) — 기술적 최우선 과제

  4순위: **에러 페이지 (404/500) 구현** (3명 공통 지적) — Quick Win, 30분 내 수정 가능
