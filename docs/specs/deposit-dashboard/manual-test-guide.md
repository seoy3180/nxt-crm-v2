> 시드 적용 상태: 3개 계좌 (critical/warning/ok)
> 작성: 2026-05-15 (Phase 7)

## 1) 로그인 + 사이드바 확인

1. http://localhost:3001 접속 → 로그인 (admin 또는 MSP 팀 계정 권장)
2. **사이드바 좌측 "MSP" 섹션 끝**에 **"예치금"** 메뉴 확인
3. 메뉴 우측에 **빨간 원형 배지 "1"** (critical 계좌 1건) 노출

체크리스트:
- [ ] 사이드바 메뉴 "예치금" 표시됨
- [ ] lucide `wallet` 아이콘 표시됨
- [ ] 빨간 배지 "1" 표시됨 (critical 1건)
- [ ] 비-MSP 팀 계정으로 로그인 시 메뉴 자체 미노출 (Education/Dev 팀)

## 2) 메인 대시보드 (/deposit)

사이드바 "예치금" 클릭 → `/deposit` 진입.

### 글로벌 KPI 4박스
- [ ] **총 예치액**: `₩ 20,000,000` (3건 합계) + "USD 0건 · KRW 3건"
- [ ] **누적 사용액**: `₩ 10,970,000` + 예치 대비 비율 표기
- [ ] **현재 잔액** (emerald 박스): `₩ 9,030,000`
- [ ] **알림 필요** (red 박스): `1 긴급` / `1 주의` (통화 무관 합산)

### 필터 + 카드 그리드
- [ ] 필터 버튼: 전체(3) / 긴급(1) / 주의(1)
- [ ] 카드 정렬 순서:
  1. 코난테크 (critical, 빨간 테두리)
  2. 디스코 (warning, 호박색 테두리)
  3. 서울대 (ok, 초록 테두리)
- [ ] 필터 "긴급" 클릭 → 코난테크 카드만 표시
- [ ] 카드별 정보:
  - 헤더 (계약명 + 고객명 + 우측 배지)
  - 현재 잔액 강조 영역 (alertLevel 색상)
  - 총 예치/누적 차감 2박스
  - 월평균 소진 + 소진 예상일
  - 최근 입출 4건
  - + 예치 등록 / − 사용 차감 버튼

## 3) 거래 등록 (모달)

서울대(ok) 카드의 "+ 예치 등록" 클릭.

- [ ] 모달 헤더: "예치 등록" + "KRW 통화 계좌"
- [ ] 일자 default = 오늘
- [ ] 금액 1,000,000 + 메모 "수동 테스트 입금" 입력 → 등록
- [ ] 토스트 "예치 1,000,000이 등록되었습니다" 표시
- [ ] 카드 잔액이 `₩ 8,950,000`으로 즉시 갱신
- [ ] 거래내역에 새 행 등장

**음수 잔액 2차 확인 (BL-1):**
1. 코난테크 카드의 "− 사용 차감" → 100,000 입력 → 등록
2. 모달 내 amber 박스: "차감 후 잔액이 음수가 됩니다. 다시 등록을 클릭..."
3. 다시 등록 클릭 → 진행
4. 잔액이 `−120,000`으로 갱신

체크리스트:
- [ ] 음수 잔액 2차 확인 모달 동작
- [ ] 차감 진행 후 카드 알림이 critical 유지

- [ ] admin 계정: "잔액 보정" / "환불 등록" 버튼 추가 노출

체크리스트:
- [ ] 비활성화 차단 메시지 표시

## 5) 비활성 상태 (활성화 흐름)

다른 MSP 계약 (deposit 시드 없는 계약, 전체 57건 중 4번째 이후) 상세 진입.

- [ ] 예치금 영역에 **"활성화" 버튼** + 안내 텍스트
- [ ] "예치금 계좌 활성화" 클릭 → 확인 모달
- [ ] 활성화 후 Empty State 카드 표시 (잔액 0)

### 활성화 직후 비활성화 (트랜잭션 0건)
- [ ] 비활성화 모달이 정상 동작 (트랜잭션 없으니 차단 X)
- [ ] 클릭하면 잔액 0인 빈 계좌 비활성화됨 (deleted_at)

## 6) 계약 삭제 차단 (BIZ-5)

코난테크(잔액 -20,000) 또는 디스코(잔액 1,100,000) 상세 → 우측 메뉴 → "계약 삭제" 시도.

- [ ] **삭제 차단 토스트**: "예치금 잔액이 ₩ XXX 남아있어 삭제할 수 없습니다"
- [ ] 계약은 `deleted_at` 변경 없이 그대로 유지

수동 복구 시도:
- 잔액 0인 서울대를 admin으로 환불(refund) 7,950,000 입력 → 잔액 0
- 계약 삭제 → 정상 삭제 → 사이드바 카운트 갱신
- ⚠️ **이 시나리오 실행 시 contracts.deleted_at 변경됨 → rollback SQL의 (3) 라인 실행 필요**

## 7) 무효화 (void) 흐름

거래내역 테이블에서 입력된 트랜잭션 행 우측 "무효화" 클릭.

- [ ] 모달 표시 + 사유 5자 이상 요구
- [ ] 사유 입력 후 확인 → 잔액 자동 재계산 (트리거)
- [ ] 거래내역 행이 **취소선 + 50% 흐림**
- [ ] 같은 행에 "무효화" 버튼 더 이상 노출 X

## 핵심 회귀 포인트

테스트 종료 후 반드시:

1. **rollback 실행** (사용자 승인 후):
   ```sql
   TRUNCATE deposit_transactions, deposit_accounts CASCADE;
   ```

2. **계약 삭제 시나리오 실행한 경우** — 해당 contracts.deleted_at 복구:
   ```sql
   UPDATE contracts SET deleted_at = NULL WHERE id = '<삭제한_계약_id>';
   ```

3. **검증**:
   ```sql
   SELECT COUNT(*) FROM deposit_accounts;       -- 0
   SELECT COUNT(*) FROM deposit_transactions;   -- 0
   SELECT COUNT(*) FROM contracts WHERE deleted_at IS NOT NULL;  -- 테스트 전과 동일 카운트
   ```
